The Flaky Network: Final Architecture & Engineering Design

Mission: Transport a live, 160-byte 50fps audio stream across a highly volatile UDP relay network while surviving drop spikes, reordering, and heavy jitter. Must achieve ≤ 1.00% packet miss rate while mathematically guaranteeing a strict bandwidth overhead of ≤ 2.00x.

1. Mathematical Rejection of Naive Strategies

Before writing code, we evaluated standard transport paradigms against the strict physical constraints of the assignment:

  • Why TCP / NACK Retransmission Fails: A lost packet detected at the receiver requires a NACK to travel back through the hostile relay, and the retransmitted packet to traverse the relay a third time. At 50fps (20ms per frame), this triple round-trip delay mathematically guarantees a deadline miss on volatile profiles.
  • Why RFC-2198 Piggybacking Fails: Sending the current frame (160 bytes) and a redundant copy of the previous frame (160 bytes) costs exactly 320 bytes. The 2.0x assignment budget is precisely 328 bytes. The moment we add a 4-byte Sequence ID and 1-byte Packet Type, the packet inflates to 325+ bytes. This pushes bandwidth overhead to ~2.03x, which results in an instant INVALID score.

2. Chosen Architecture: Adaptive Interleaved Forward Error Correction (FEC)

The Solution: A proactive XOR parity engine (K=2). For every 2 data frames, the C++ sender computes and emits exactly 1 parity packet (P = A ⊕ B).

Because it operates at a 3:2 ratio, the bandwidth overhead is precisely 1.54x. This perfectly solves the 2.00x constraint while leaving a massive bandwidth buffer for reverse-telemetry.

Defeating Burst Drops via Stride Interleaving

Network drops on Profile B are rarely isolated; they arrive in consecutive bursts. A naive parity engine (Pairing Frame 0 and Frame 1) fails if both frames are dropped in a micro-burst. We solved this via Stride Interleaving. By setting Stride=2, Frame 0 pairs mathematically with Frame 2. A 2-packet burst dropping Frame 1 and Frame 2 segregates the losses into entirely different parity groups, ensuring 100% mathematical recovery.

3. Closed-Loop Telemetry & Adaptive Jitter Control

High strides provide ultimate burst protection, but require a larger delay_ms jitter buffer to wait for the distant parity partner to arrive. Rather than penalizing clean networks with high latency, we engineered a closed-loop telemetry system.

The C++ Receiver maintains a 500ms sliding window, analyzing the exact network burst-loss pattern. Every 500ms, it transmits an ultra-lightweight 8-byte telemetry packet back to the sender. If the network is pristine, the sender dynamically shifts to Stride 1, maximizing available jitter-buffer headroom. If a burst is detected, it shifts to Stride 2 to weather the storm.

4. C++ Implementation Details & Wire Format

The sender and receiver are engineered in pure C++17. They utilize fcntl(O_NONBLOCK) and strict select() event loops to guarantee the packet intake and processing pipeline is never blocked by socket I/O delays. Memory allocation is static (zero-allocation main loop) to prevent GC pauses.
