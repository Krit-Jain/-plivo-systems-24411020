The Flaky Network: FEC Architecture Summary

This document details the architectural decisions and implementation strategy for building a robust real-time media transport system across a highly volatile UDP relay network.

1. The Problem Space & Constraints

The core challenge is delivering 160-byte audio frames every 20ms over a network that introduces random drops, bursts of drops, high jitter, and reordering. The rigid constraints are:

  • Strict Deadline: Frames must arrive by T0 + delay_ms + i*20ms. Late frames are identical to dropped frames.
  • Bandwidth Budget: Total network traffic (including both directions) cannot exceed 2.0x the raw stream payload.
  • Miss Rate: The final playout miss rate must be ≤ 1.00%.

2. Why Traditional Retransmission Fails

In a typical TCP-style system, a lost packet triggers a Negative Acknowledgment (NACK) from the receiver, prompting the sender to resend the data. However, in our environment, the time required for a frame to travel across the network, fail, trigger a NACK, and have the resend travel back across the network far exceeds the 20ms real-time deadline. Furthermore, retransmissions inherently cause head-of-line blocking. Real-time constraints demand proactive solutions.

3. Forward Error Correction (FEC): Interleaved XOR

Instead of reacting to loss, we proactively send redundant data. We implemented an Interleaved XOR Parity system.

The XOR Parity Mechanism (K=2)
For every two data frames (e.g., Frame A and Frame B), the sender computes a parity packet: Parity = Frame A ⊕ Frame B.
If Frame A is lost in transit, the receiver can instantly reconstruct it by computing: Frame A = Frame B ⊕ Parity.

This specific configuration costs exactly 1.5x bandwidth overhead (sending 3 packets for every 2 frames of data), which fits perfectly within our 2.0x strict budget, leaving ample room for protocol headers and feedback telemetry.

4. Defeating Burst Loss: Stride Interleaving

A fatal weakness of simple parity is burst loss. If a network outage drops two packets in a row, both Frame A and Frame B might be lost, rendering the Parity packet useless. To solve this, we implemented Stride Interleaving.

By increasing the stride (e.g., Stride = 2), the sender pairs non-adjacent frames. Frame 0 is paired with Frame 2, and Frame 1 is paired with Frame 3. If a network burst drops frames 1 and 2, they belong to different parity groups and can both be mathematically recovered! The tradeoff is delay: the sender must hold Frame 0 in a buffer until Frame 2 arrives 40ms later to compute the parity.

5. The Adaptive Feedback Loop

Because higher strides mandate a larger jitter buffer delay, we want to use the lowest stride possible unless the network is actively hostile. We built a telemetry loop using the optional feedback path.

  • Receiver Telemetry: Every 25 frames, the receiver analyzes the arrival history to calculate the maximum consecutive loss run (the "burst length"). It sends this data back to the sender in a tiny 8-byte UDP packet.
  • Sender Adaptation: If the sender detects a clean network, it operates at Stride 1 (ultra-low delay). If the feedback indicates burst losses of 2 or more, it dynamically shifts its mathematical grouping to Stride 2 to weather the storm.

6. System Level Integration

At the system level, the sender and receiver are written in highly optimized C++17 utilizing non-blocking sockets (O_NONBLOCK) and select() event loops. This ensures that the processes never stall on network I/O. The receiver operates with a "Zero-Delay Forwarding" policy: incoming data frames are instantly pushed to the harness player, while parity calculations are handled asynchronously in the background. The final wire protocol is precisely 164 bytes (a 4-byte custom header appended to the 160-byte raw payload), ensuring maximal efficiency and zero fragmentation.
